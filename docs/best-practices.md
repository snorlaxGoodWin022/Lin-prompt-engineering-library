# Prompt 工程最佳实践

基于原始文档和实际使用经验总结的 Prompt 工程最佳实践。

## 核心原则

### 1. 说清楚你要什么（明确需求）
- **具体 vs 模糊**：不要只说"帮我做个页面"，要说"帮我做个用户管理页面，包含列表、搜索、新增、编辑功能"
- **提供上下文**：说明项目背景、用户场景、技术约束
- **定义成功标准**：什么样的输出算是成功的

### 2. 告诉AI你的项目技术栈
- 框架和版本：React 18、Vue 3.4
- UI库：Ant Design 5.x、Element Plus
- 状态管理：Zustand、Pinia
- 样式方案：CSS Modules、SCSS、Tailwind
- 构建工具：Webpack 5、Vite

### 3. 给出具体的代码规范要求
- 命名规范：PascalCase、camelCase、kebab-case
- 文件结构：组件文件夹组织
- 编码规则：TypeScript类型、注释要求、错误处理
- 质量要求：测试覆盖率、性能指标

### 4. 提供输入输出示例
- 好的示例：展示期望的输出格式和质量
- 不好的示例：指出要避免的问题
- 边界案例：特殊情况的处理方式

## 实用技巧

### 技巧1：分步骤拆解复杂需求

**不好的Prompt：**
```
帮我做一个用户管理页面
```

**好的Prompt：**
```
我要做一个用户管理页面，分三步：

第一步：先帮我设计数据结构和接口
- 用户列表接口
- 用户详情接口
- 创建/编辑/删除接口

第二步：根据接口设计页面组件结构
- 列表组件
- 搜索筛选组件
- 编辑弹窗组件

第三步：写具体代码
- 使用React + Ant Design
- 状态管理用Zustand
- 表单校验用react-hook-form

现在先做第一步，设计好了我再让你做第二步。
```

### 技巧2：提供正反示例

**示例Prompt：**
```
帮我写一个表单验证函数。

【好的示例】
// 清晰的错误提示，指出具体问题
if (!email.includes('@')) {
  return { valid: false, message: '邮箱格式不正确，缺少@符号' };
}

【不好的示例】
// 模糊的错误提示
if (!validate(email)) {
  return { valid: false, message: '输入错误' };
}

按照好的示例风格，给我写手机号、身份证号的验证函数。
```

### 技巧3：明确输出格式

**示例Prompt：**
```
帮我整理这个API接口文档。

【输入】
{粘贴杂乱的接口信息}

【输出格式要求】
用Markdown表格，必须包含：
| 字段名 | 类型 | 必填 | 说明 | 示例值 |

不要给我文字描述，直接给表格，我要复制粘贴到文档里。
```

### 技巧4：使用约束性语言
- **必须**：强制性要求，如"必须使用TypeScript"
- **禁止**：明确不允许的做法，如"禁止使用any类型"
- **至少**：最低要求，如"至少写3个测试用例"
- **优先**：偏好选择，如"优先使用函数组件"

### 技巧5：迭代优化Prompt
1. **记录问题**：AI生成的代码哪里不符合预期
2. **调整Prompt**：补充缺失的规范说明
3. **测试验证**：用新Prompt再生成一次
4. **版本管理**：Prompt也要git管理，记录改动原因

## 模板使用技巧

### 1. 模板定制化
- 根据项目实际情况调整技术栈
- 添加团队特有的编码规范
- 考虑项目历史和技术债务

### 2. 模板组合使用
- 组件生成模板 + 文档生成模板
- 性能分析模板 + 优化方案模板
- 代码生成模板 + 测试生成模板

### 3. 模板版本管理
```
v1.0 - 基础React组件Prompt，但AI经常忘记写PropTypes
v1.1 - 补充"必须写完整TypeScript类型"要求
v1.2 - 增加"导出类型定义"要求
v2.0 - 加入错误边界和loading处理要求
```

## 常见问题解决

### Q1：AI生成的代码不符合项目规范怎么办？
- 在Prompt中明确项目规范
- 提供项目中的代码示例作为参考
- 使用更具体的约束语言

### Q2：同样的Prompt，有时效果好有时不好？
- AI有随机性，多生成几次取最好的
- 降低温度参数（如果有）
- 使用更确定性的描述

### Q3：Prompt需要经常更新吗？
- 技术栈升级 → 必须更新
- 新增代码规范 → 及时补充
- 发现AI常犯错误 → 针对性优化

### Q4：如何评估Prompt效果？
- 生成的代码能否直接使用
- 需要手动修改的程度
- 团队成员的接受度
- 实际开发效率提升

## 总结

好的Prompt工程是：
- **系统性**的思考，不是临时起意
- **迭代式**的优化，不是一次完成
- **团队性**的实践，不是个人技巧
- **数据驱动**的改进，不是主观感觉
