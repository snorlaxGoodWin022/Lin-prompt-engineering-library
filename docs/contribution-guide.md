# 贡献指南

欢迎为 Prompt 工程库贡献新的模板、示例或改进！

## 贡献类型

### 1. 新 Prompt 模板
- 针对新的开发场景
- 现有模板的变体或优化
- 新的技术栈支持

### 2. 使用示例
- 模板的实际使用示例
- 生成效果评估
- 最佳实践案例

### 3. 工具和脚本
- 模板验证工具
- 格式化工具
- 辅助使用工具

### 4. 文档改进
- 使用指南
- 最佳实践
- 教程和案例

### 5. Bug 修复和改进
- 模板错误修正
- 格式改进
- 内容更新

## 贡献流程

### 第一步：准备工作
1. Fork 本仓库
2. 克隆到本地：`git clone git@github.com:your-username/prompt-engineering-library.git`
3. 创建新分支：`git checkout -b feature/your-feature-name`

### 第二步：开发贡献
#### 添加新 Prompt 模板
1. 选择合适的分类目录
2. 创建新的模板文件，使用 `.md` 扩展名
3. 遵循现有模板的结构和格式
4. 在对应目录的 README.md 中更新模板列表
5. 添加使用示例（如可能）

#### 模板文件规范
```markdown
# 模板标题

{模板描述和说明}

## 部分标题
{具体内容}

## 使用示例
```markdown
{完整的示例使用方式}
```

## 模板说明
{设计思路、适用场景、注意事项}
```

#### 文件命名规范
- 使用英文小写文件名
- 单词用连字符分隔：`react-component.md`
- 避免空格和特殊字符
- 保持名称描述性

### 第三步：测试验证
1. 测试模板的实际使用效果
2. 验证生成代码的质量
3. 确保模板清晰、完整
4. 检查占位符和格式

### 第四步：提交贡献
1. 添加文件：`git add .`
2. 提交更改：`git commit -m "Add: 描述你的贡献"`
3. 推送分支：`git push origin feature/your-feature-name`
4. 创建 Pull Request

### 第五步：Pull Request 要求
#### PR 标题格式
- `Add: 新模板名称` - 添加新模板
- `Update: 模板名称` - 更新现有模板
- `Fix: 问题描述` - 修复问题
- `Docs: 文档更新` - 文档改进

#### PR 描述要求
1. **改动说明**：详细说明你的贡献
2. **动机和背景**：为什么需要这个贡献
3. **测试情况**：如何测试和验证
4. **相关链接**：Issue、讨论等
5. **截图/示例**：展示效果

## 模板质量标准

### 内容质量
- **完整性**：包含所有必要部分
- **清晰性**：语言清晰，结构合理
- **实用性**：解决实际开发问题
- **一致性**：遵循现有模板风格

### 技术质量
- **准确性**：技术栈、规范准确
- **可测试性**：可以验证使用效果
- **可维护性**：易于更新和维护
- **兼容性**：考虑不同使用场景

### 文档质量
- **使用说明**：清晰的用法说明
- **示例完整**：提供完整的使用示例
- **注意事项**：说明限制和特殊考虑
- **版本记录**：重要变更记录

## 审核标准

### 会通过审核的贡献
1. 解决实际问题或填补空白
2. 遵循项目规范和标准
3. 提供完整的示例和测试
4. 文档清晰、完整
5. 代码/内容质量高

### 可能需要修改的贡献
1. 格式不符合项目规范
2. 缺少必要的文档或示例
3. 技术栈过时或不准确
4. 与其他模板重复度高
5. 使用场景过于狭窄

### 可能被拒绝的贡献
1. 包含敏感或不当内容
2. 技术上有明显错误
3. 质量不符合项目标准
4. 与项目目标不符
5. 存在版权或法律问题

## 行为准则

### 基本原则
1. **尊重**：尊重其他贡献者和用户
2. **建设性**：提供建设性的反馈和建议
3. **包容性**：欢迎不同背景和技能的贡献者
4. **专业性**：保持专业和友好的交流

### 沟通指南
- 使用清晰、礼貌的语言
- 提供具体的反馈和建议
- 积极回应问题和讨论
- 保持耐心和理解
